# How html.to.design converts websites into editable Figma files

**html.to.design uses a dual-architecture system — a Chrome extension leveraging the Chrome DevTools Protocol and a server-side headless Chromium renderer — to capture complete DOM snapshots with resolved computed styles, serialize them into a proprietary `.h2d` intermediate format, and reconstruct pixel-accurate Figma node trees via the Plugin API.** Built by the EU-based company ‹div›RIOTS, the tool has attracted over 1.3 million users by solving one of design tooling's hardest problems: faithfully translating the browser's rendering engine output into editable vector design primitives. The underlying engine, called **code.to.design**, is also exposed as a standalone API for third-party plugin developers. This report dissects every layer of the technical pipeline — from DOM capture through design token extraction to Figma node construction — and maps the open-source landscape for anyone building their own implementation.

## The two-component architecture captures any page state

html.to.design operates as **both a Figma plugin and a Chrome extension**, each serving a distinct capture pathway that feeds the same conversion engine.

**Path 1 — Figma Plugin (public URLs):** When a user pastes a URL into the plugin, the request is sent to code.to.design's backend API servers, which render the page in a headless Chromium instance. This server-side approach handles full JavaScript execution, viewport-size selection (desktop, tablet, mobile), and light/dark theme toggling. The server performs DOM traversal, computed style extraction, and asset bundling, then returns a structured intermediate representation to the plugin's client-side SDK, which constructs native Figma nodes on the canvas.

**Path 2 — Chrome Extension (private/authenticated pages):** For pages behind login walls, on private networks, or in specific interactive states, the companion Chrome extension (ID: `ldnheaepmnmbjjjahokphckbpgciiaed`) captures the page directly from the user's browser. Critically, this extension **requires the `chrome.debugger` permission**, which grants access to the full Chrome DevTools Protocol (CDP). The Chrome Web Store listing states explicitly: *"In order to best map what you see in your browser to Figma, we need to access Chrome's debugging features."* This is far more powerful than simple content script injection — CDP provides access to the `DOMSnapshot` domain (capturing the entire DOM tree with computed styles in a single call), the `CSS` domain (matched rules and computed values), the `Page` domain (screenshots and resource content), and the `Network` domain (intercepting loaded images, fonts, and stylesheets).

The extension packages captured data into a **proprietary `.h2d` file** — a serialized snapshot containing HTML markup, resolved styles, layout metadata, and asset references that mirrors the DOM structure with additional conversion-friendly metadata. This file can be sent directly to the Figma plugin or downloaded locally for offline use. The code.to.design API then processes `.h2d` files through its SDK to generate Figma nodes, completing the pipeline.

## From getComputedStyle to Figma nodes: the conversion pipeline

The core extraction technique relies on `window.getComputedStyle()` — the browser API that returns **resolved values** of all CSS properties after applying stylesheets, inheritance, cascading, and computation. This is the critical insight that makes browser-based conversion vastly superior to static CSS parsing: `getComputedStyle` automatically resolves CSS variables to final values, converts all relative units (em, rem, vh, vw, %, calc()) to absolute pixels for layout properties, and returns the actual rendered state including CSS-in-JS styles that never exist in any stylesheet.

The pipeline operates in seven stages:

**Stage 1 — DOM traversal.** A recursive walker traverses the DOM tree starting from the document body. For each visible element, the system calls `getComputedStyle()` to extract all resolved CSS properties and `getBoundingClientRect()` to capture exact position and dimensions. Pseudo-elements are explicitly checked via `getComputedStyle(element, '::before')` and `getComputedStyle(element, '::after')`. **Stage 2 — Shadow DOM piercing.** Modern component-based sites use Web Components with shadow DOM boundaries that hide internal structure from normal DOM traversal. The tool pierces these boundaries to capture the complete visual tree. **Stage 3 — Asset collection.** Images (from `<img>` tags, background-image URLs, and inline SVGs), web fonts (from `@font-face` rules and the `document.fonts` API), and other resources are collected and bundled. SVGs are serialized via `new XMLSerializer().serializeToString()`. **Stage 4 — Iframe resolution.** Content embedded in iframes has separate DOMs and potentially separate origins, requiring recursive capture. **Stage 5 — Serialization to `.h2d` format.** The extracted DOM tree, computed styles, asset references, and layout data are serialized into the intermediate representation. **Stage 6 — CSS-to-Figma property translation.** The conversion engine maps each CSS property to its Figma equivalent. **Stage 7 — Figma node construction.** The Plugin API SDK creates native Figma nodes, preserving the parent-child hierarchy as Figma layer nesting.

The html.to.design blog enumerates the specific technical challenges they solved: *"box-sizing, stacking context, the extraction of styles, DOM serialization, shadow DOM piercing, proper rendering of fonts, optimal asset bundling and iframe nesting."*

## CSS property mapping to Figma: where precision meets compromise

The mapping from CSS properties to Figma node properties is extensive but not always one-to-one. Some mappings are direct; others require significant transformation.

**CSS Flexbox maps cleanly to Figma Auto Layout.** `display: flex` becomes a FrameNode with `layoutMode` set to `'HORIZONTAL'` or `'VERTICAL'` based on `flex-direction`. The `gap` property maps to `itemSpacing`, `justify-content` maps to `primaryAxisAlignItems` ('MIN', 'CENTER', 'MAX', 'SPACE_BETWEEN'), and `align-items` maps to `counterAxisAlignItems`. Child properties like `flex-grow: 1` become `layoutGrow = 1`, and `flex-wrap: wrap` becomes `layoutWrap = 'WRAP'`. Sizing modes translate as: child with `width: auto` → `layoutSizingHorizontal = 'HUG'`, child filling available space → `'FILL'`, and explicit pixel width → `'FIXED'`.

**Colors require normalization.** CSS returns colors as `rgb()` or `rgba()` strings; Figma expects `{r, g, b}` objects normalized to the 0–1 range (divide each channel by 255). A CSS `background-color` becomes a `fills` array entry of type `'SOLID'`. Background gradients require more complex transformation — CSS uses angle-based notation (e.g., `135deg`) while Figma uses a `gradientTransform` affine matrix with three 2D handle positions, requiring trigonometric conversion.

**Shadows translate with one caveat.** `box-shadow: 10px 10px 20px 5px rgba(0,0,0,0.25)` maps directly to a `DROP_SHADOW` effect with `offset: {x: 10, y: 10}`, `radius: 20`, `spread: 5`, and the parsed color. The `inset` keyword changes the type to `INNER_SHADOW`. CSS `filter: blur()` becomes `LAYER_BLUR`, and `backdrop-filter: blur()` becomes `BACKGROUND_BLUR`. The caveat: Figma renders shadows through transparent fills (physically-based rendering), while CSS masks shadows behind geometry, so transparent elements may look different.

**Margins have no Figma equivalent.** This is one of the hardest conversion challenges. Open-source tools like BuilderIO's html-to-figma handle this by creating **wrapper frames with padding** equal to the margin values, then nesting the element inside. Alternatively, margins between flex children can be approximated with the parent's `itemSpacing`.

**Text requires async font loading.** Every text property modification in Figma requires a prior call to `figma.loadFontAsync({ family, style })`. CSS `font-weight` (a numeric value like 700) must be mapped to Figma's font style string ("Bold"), which varies by typeface. Line-height maps to `{value, unit: 'PIXELS'}` for pixel values, `{unit: 'AUTO'}` for CSS `normal`, and `{value, unit: 'PERCENT'}` for percentage values.

**Positioning requires careful translation.** CSS `position: absolute` children within flex containers map to `layoutPositioning = 'ABSOLUTE'` within auto-layout frames (supported since 2023). `overflow: hidden` maps to `clipsContent = true`. CSS `z-index` has no direct property — it controls **layer ordering** within the Figma children array, requiring z-index-based sorting during conversion.

## The Figma Plugin API's sandbox model shapes everything

The Figma Plugin API operates on a **strict two-thread architecture** that fundamentally constrains how conversion tools work. The **main thread** runs in a sandboxed JavaScript environment (QuickJS compiled to WebAssembly) with access to the `figma` global object for scene manipulation but **zero browser APIs** — no DOM, no `fetch`, no `localStorage`. The **UI thread** runs in an iframe with full browser capabilities but cannot touch the Figma scene. Communication between threads is exclusively via `postMessage` with structured clone serialization.

This means all network requests (fetching pages, downloading images, calling APIs) must happen in the UI iframe, while all Figma node creation must happen in the main thread. Data flows UI → main thread via `parent.postMessage({pluginMessage: data}, '*')`. The `networkAccess.allowedDomains` field in `manifest.json` restricts which domains the UI iframe can contact.

For design token creation, the **Variables API** (`figma.variables`) enables plugins to create Variable Collections with multiple modes (e.g., Light/Dark), create typed variables (COLOR, FLOAT, STRING, BOOLEAN), set values per mode, and bind variables to node properties. The **Styles API** creates reusable Paint Styles, Text Styles, and Effect Styles that can be applied across the document. html.to.design automatically generates local Figma styles from extracted text and color patterns, creating an instant design system from any website.

Key creation methods include `figma.createFrame()` for containers, `figma.createRectangle()` for solid shapes, `figma.createText()` for text nodes (requiring async font loads), `figma.createNodeFromSvg(svgString)` for SVG import, and `figma.createImage(uint8Array)` for image creation from raw bytes. Images aren't standalone nodes — they're applied as `IMAGE` type fills on shapes.

## Open-source alternatives reveal the implementation patterns

Several open-source projects implement parts of the html-to-design pipeline, providing concrete code examples for anyone building their own tool.

**BuilderIO/figma-html** (3,500 GitHub stars, MIT license) is the most complete open-source reference. Its `@builder.io/html-to-figma` npm package exposes `htmlToFigma(document.body)`, which recursively traverses the DOM, calls `getComputedStyle()` on each element, and produces an intermediate JSON representation of Figma-compatible layer nodes. The library defines a `LayerNode` type system with properties for geometry, fills, strokes, effects, and layout. Its two-phase architecture — browser-side extraction → JSON → Figma plugin-side construction — mirrors html.to.design's approach. Builder.io reports **80–90% accuracy** on typical pages; html.to.design users report approximately 97%.

**Dembrandt** (1,500 stars, MIT license) takes a different approach, focused on **design token extraction** rather than full layout reproduction. It uses Playwright (not Puppeteer) for headless browser automation with anti-bot-detection stealth configuration. The extraction pipeline runs JavaScript directly in the rendered page to call `getComputedStyle()`, scan CSS custom properties on `:root`, analyze color usage patterns with confidence scoring, group typography patterns, detect spacing scales from margin/padding values, and extract component patterns (buttons, inputs, badges). Output follows the **W3C Design Tokens Community Group (DTCG) specification**. It supports dark mode extraction (`--dark-mode`), mobile viewports (`--mobile`), and multiple output formats (JSON, DTCG, terminal display).

**Project Wallace's ecosystem** provides a complementary static analysis pipeline: `extract-css-core` uses a Chromium instance to collect all CSS from a URL (including CSS-in-JS), `css-analyzer` (353 stars) performs comprehensive CSS analysis, and `@projectwallace/css-design-tokens` extracts W3C-spec-compliant tokens from raw CSS using AST parsing. This pipeline cannot resolve computed values but is fast and deterministic.

**Amazon Style Dictionary** (~4,000 stars) is not an extraction tool but the **de facto standard for token transformation** — it takes design token JSON files and outputs platform-specific formats (CSS variables, SCSS, iOS Swift, Android XML, JavaScript modules). Its transform pipeline (parse → merge → transform → format → build) is the backbone of most production design token workflows.

## Building your own pipeline: key patterns and edge cases

For someone implementing their own HTML-to-design-token extraction, the architecture follows three proven patterns.

**Pattern A (in-browser DOM traversal)** is the most faithful approach. Inject a script into the page, traverse the DOM recursively, call `getComputedStyle()` per element plus `getBoundingClientRect()` for geometry, and serialize to an intermediate JSON format. This captures the fully rendered state including CSS-in-JS, dynamic content, and resolved custom properties. The key libraries: a Chrome extension or content script for injection, structured clone for serialization, and the Figma Plugin API for reconstruction.

**Pattern B (headless browser extraction)** works for any URL without requiring a browser extension. Use Playwright or Puppeteer to render the page, inject extraction scripts, and capture the same `getComputedStyle` data. Dembrandt's implementation shows how to add anti-bot-detection, retry logic, and multi-viewport support. The trade-off is speed — headless browser startup adds latency.

**Pattern C (static CSS analysis)** parses raw CSS text via an AST parser without any browser context. Project Wallace's tools demonstrate this approach. It's fast but cannot resolve computed values, custom properties, or cascaded styles.

The hardest edge cases to handle are: **pseudo-elements** (not real DOM nodes; must explicitly query `getComputedStyle(el, '::before')` and create additional Figma layers), **CSS variables** (`getComputedStyle` resolves them automatically, but preserving the variable names for design tokens requires separate CSSOM parsing), **web fonts** (detect via `document.fonts` API and `@font-face` rules; CORS may block cross-origin stylesheet access; Figma requires `loadFontAsync` and missing fonts need fallback mapping), **CSS Grid** (Figma only recently added Grid auto-layout in beta at Config 2025; previously required approximation with nested auto-layout frames), **CSS transforms** (simple rotation maps to `node.rotation`; complex matrix transforms may need the `relativeTransform` property or rasterization), and **animations** (only the static snapshot at extraction time is captured; no tool currently converts CSS animations to Figma prototyping interactions).

## Conclusion

The html.to.design pipeline reveals that pixel-perfect HTML-to-Figma conversion is fundamentally a **browser rendering problem**, not a CSS parsing problem. The decisive technical choice is using `getComputedStyle` (or CDP's DOMSnapshot) to capture the browser engine's *already-resolved* rendering output, sidestepping the enormous complexity of reimplementing CSS cascading, inheritance, and layout. The `.h2d` intermediate format acts as a portable DOM snapshot that decouples capture from construction. On the Figma side, the Plugin API's auto-layout system maps surprisingly well to CSS flexbox, but CSS margins, grid layouts, pseudo-elements, and complex transforms remain genuine gaps requiring creative workarounds. For builders, the open-source landscape offers production-ready components: BuilderIO/figma-html for the full DOM-to-Figma pipeline, Dembrandt for design token extraction with anti-bot resilience, and Style Dictionary for cross-platform token transformation. The key architectural insight is that this is a two-phase system — browser-side capture and design-tool-side reconstruction — connected by a well-structured intermediate representation, and getting that intermediate format right is what separates 80% accuracy from 97%.
